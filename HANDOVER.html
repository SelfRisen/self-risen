1 Executive Summary

What the System Does

Self-Risen is a personal growth and self-development platform. The backend API powers a mobile application that helps users transform limiting beliefs into positive affirmations through AI-powered reflection sessions, track life balance with a Wheel of Life assessment tool, create vision boards, and maintain journals. The platform uses OpenAI for audio transcription, natural language processing to reframe beliefs, and text-to-speech with six distinct voice personas for personalized affirmation playback.

Current Scope

  • Reflection Sessions — Full lifecycle: audio upload → transcription (Whisper) → NLP affirmation generation (GPT-3.5) → TTS audio (6 voice personas) → wave-based listening periods
  • Wheel of Life — Category-based life assessment with scoring, imbalance detection, and focus areas
  • Vision Boards — Per-category and global boards with image uploads and background sounds
  • Journaling — CRUD with image attachments
  • Multi-channel Notifications — Email (Mailjet/Gmail/Mailgun), SMS (Twilio), Push (Expo), In-App via Bull queue
  • Streak Tracking — Daily usage streaks with timezone-aware reminders
  • Token Usage Limiting — Monthly quota system for AI token consumption
  • Authentication — Firebase Auth with Google, Apple, and Facebook OAuth

Out of Scope

  • Admin dashboard / CMS
  • Payment processing / subscriptions
  • Real-time collaborative features (WebSocket gateway is present but not broadly used)

Current Status

Stable Areas

Auth flows, Reflection sessions, Wheel of Life, Journal, Vision Boards, Notification dispatch pipeline

Monitor

OpenAI API usage / token limits, session expiration cron jobs, streak reminder scheduling

⚠
Top 5 Things a New Dev Must Not Break
  1. Reflection session state machine (PENDING → BELIEF_CAPTURED → AFFIRMATION_GENERATED → COMPLETED) — any break here corrupts user sessions.
  2. Firebase authentication guard — protects every user-facing endpoint.
  3. Prisma migrations — always run prisma migrate deploy before startup; never edit existing migrations.
  4. OpenAI integration pipeline (transcription → NLP → TTS) — the core value proposition.
  5. Notification Bull queue — failed jobs go to DLQ; retry policy must remain exponential backoff.

Architectural Decisions

  • Modular monolith (not microservices) — all modules live in a single NestJS app. Clean module boundaries enable future extraction if needed.
  • Sync API + async queue — HTTP request/response for most operations; Bull queue for notification dispatch with retry + DLQ.
  • Firebase-first auth — leverages Firebase for identity management, with custom JWT as fallback.
  • Dual storage providers — Firebase Storage and Supabase are interchangeable via STORAGE_PROVIDER env var.
  • Prisma ORM — type-safe database access, auto-generated client, migration-first schema changes.
  • OpenAI pipeline — sequential: Whisper transcription → GPT-3.5 affirmation reframing → TTS-1 voice synthesis.

Common First-Day Problems

⚠
Prisma generate fails: Make sure PostgreSQL is running and DATABASE_URL is set correctly. Run pnpm prisma generate before pnpm start:dev.
⚠
Redis connection refused: Ensure Redis is running on port 6379. Check REDIS_HOST and REDIS_PORT env vars.
⚠
FFmpeg not found: Install FFmpeg system-wide. On macOS: brew install ffmpeg. On Ubuntu: apt install ffmpeg. On Windows: download from ffmpeg.org and add to PATH.
ⓘ
Firebase credentials: You need a firebase-credentials.json file or the corresponding FIREBASE_* env vars set for auth to work.

9 Operations Runbooks

Service Won't Start

StepAction
SymptomsContainer exits immediately, health check fails, Railway shows "crashed"
Quick CheckCheck Railway logs for startup errors. Most common: DATABASE_URL not set, Prisma migration failure, missing env vars.
Mitigation1) Verify all required env vars in Railway. 2) Check if PostgreSQL is reachable. 3) Run prisma migrate deploy manually if migration stuck.
EscalationIf DB is unreachable → check Railway PostgreSQL service. If code error → revert last deploy.

DB Connection Exhaustion

StepAction
SymptomsPrismaClientKnownRequestError: Can't reach database server, timeouts on all endpoints
Quick CheckCheck Prisma connection pool settings. Default pool size is 10 connections.
Mitigation1) Restart the app container. 2) Add ?connection_limit=5 to DATABASE_URL if needed. 3) Check for uncommitted transactions.
Root CauseLong-running queries, unclosed connections, or too many concurrent requests for pool size.

Notification Queue Backlog

StepAction
SymptomsNotifications delayed, DLQ growing, Redis memory increasing
Quick CheckConnect to Redis and check queue length: redis-cli LLEN bull:notification_dispatch:wait
Mitigation1) Check if email/SMS providers are responding (Mailjet, Twilio status pages). 2) Restart notification consumer. 3) Check DLQ entries for error patterns.
EscalationIf provider is down → wait or switch to backup provider. If code bug → hotfix and deploy.

OpenAI API Errors / Rate Limits

StepAction
SymptomsReflection sessions stuck at PENDING or BELIEF_CAPTURED, 429 errors in logs
Quick CheckCheck OpenAI API status page. Check token usage at platform.openai.com.
Mitigation1) Rate limited → wait for reset or increase limits on OpenAI dashboard. 2) Check OPENAI_API_KEY validity. 3) If billing issue → add credits.
User ImpactUsers cannot complete reflection sessions. Existing completed sessions still work.

Elevated 5xx / Latency Spikes

StepAction
SymptomsRailway logs show 500 errors, users report slow/failed requests
Quick Check1) Check Railway deploy logs. 2) Check DB connection health. 3) Check Redis connectivity. 4) Check external API status (OpenAI, Firebase).
Mitigation1) Restart container in Railway. 2) If DB-related → check connection pool. 3) If external API → check provider status pages.
EscalationIf persistent → rollback to last known good deploy.

10 Security & Access

Authentication & Authorization

  • Identity Provider: Firebase Authentication (manages user identities)
  • API Protection: FirebaseGuard validates Firebase ID tokens on all protected routes
  • OAuth Flows: Google, Apple, Facebook sign-in supported
  • Token Revocation: Revoked token checking enabled

D) Optimization Opportunities

Cache Affirmation Audio

If users regenerate audio for the same text with the same voice, cache and reuse. Could save 30–50% of TTS costs.

Token Usage Limits

Already implemented (30K tokens/month/user). Monitor and tune the limit based on actual usage patterns.

Batch TTS Generation

Generate all affirmation audios in parallel rather than sequential to reduce wall-clock time (latency, not cost).

Image Compression Tuning

Already enabled via Sharp with 3-tier quality. Consider WebP format for smaller file sizes and lower storage costs.
